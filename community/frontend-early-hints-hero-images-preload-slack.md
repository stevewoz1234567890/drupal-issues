# Early hints + hero images + preload (Slack — `#frontend`, Apr 2026)

**Context:** Thread on whether **hero (or other) images** can be sent with **HTTP 103 Early Hints**, in the same spirit as planned work on **CSS** ([**#3563782**](https://www.drupal.org/project/drupal/issues/3563782) — early hints for CSS, depends on aggregate/library visibility).

## Takeaways

- **Responsive images vs early hints:** Choosing the “right” file for **`<img srcset>`** or **`<picture><source>`** often needs **layout / CSS / pipeline** progress. Several participants concluded that **proper** early hints for **responsive** heroes is **fundamentally constrained** in the same way as guessing the correct variant without rendering—at best a **limited** or **“non‑responsive hero”** story (e.g. a **fixed URL** like a **site logo**).
- **Link to image preload in core:** Hero-style preloading is the domain of [**#3309016 — Add image preload option**](https://www.drupal.org/node/3309016). **catch** noted that responsive preloading is discussed on [**web.dev — Preload responsive images**](https://web.dev/articles/preload-responsive-images) and that Drupal **11** work appears in [GitLab MR !14009](https://git.drupalcode.org/project/drupal/-/merge_requests/14009) (closes **#3309016**). The MR direction (as summarized in thread): **preload `<link>`** support in **responsive image** formatter, **`imagesrcset` / `imagesizes` / `media`** for sources, **`#image_preload`**, **`loading` tweaks** when preload is on—**not** a rewrite of breakpoint selection.
- **Early hints as the same transport as preload:** The same **`Link:`** hints used for **`rel=preload`** could in principle be emitted as **103** early hints, with the **same limitations** as head-based preload for responsive art direction.
- **Where to hook in Drupal (sketch):** Before **placeholder** rendering, scan for **preload-capable head attachments** (existing **`#attached`** or a dedicated key); if the **hero is not BigPipe-placeholdered**, those URLs might still be **103**-hintable **early**. After placeholders resolve, the HTML is **imminent**—**catch** guessed marginal gain (order of **~10ms**), not ruling out **multiple 103** responses during placeholder replacement as a **later** idea once the general mechanism exists.
- **Follow-up process:** Suggestion to file or extend [**#3309016**](https://www.drupal.org/node/3309016) with this analysis (possible **Won’t fix** for **responsive** early hints, or document tradeoffs). **James Wilson** offered to write it up; also **MR/issue comments** on the D11 preload work (D10 MR noted as **not** touching **responsive_image**; D11 MR **does**).
- **`sizes: auto` / “end of responsive images”:** **Lynn Stahl** linked [**Piccalilli — *The end of responsive images***](https://piccalil.li/blog/the-end-of-responsive-images/) (**Mat Marquis**). **catch**: does not remove **picture** vs **srcset** split in Drupal; **`sizes: auto`** can mean the browser still needs **element width**, which may land **after CSS**—affecting whether **early** hints buy much. **James**: **`<picture>`** branch on web.dev may complicate **Drupal**’s default **responsive image** output and **BC**.
- **Relative payoff:** Consensus tilt: **more wins** from [**#3563782**](https://www.drupal.org/project/drupal/issues/3563782) (**CSS** early hints) and **fonts** than from images; **avoid early-hinting JS**. **`fetchpriority="high"`** on **`<img>`** was mentioned as **not** yet a core focus but **fine** and complementary; **if CSS + fonts** start earlier, **images** tend to follow in the download queue anyway (with usual **connection / concurrency** caveats on slow links).
- **FOIT/FOUST / ordering:** **Fonts-first** early hints **help** perceived text stability but **do not guarantee** **no** flash if **CSS** is small and **fonts** large. **CSS vs fonts** ordering in **`Link:`** hints is worth tuning but **easy to change** later.
- **Non-responsive exceptions:** **Site logo** (often a **single URL**, **Olivero** hard-codes path—dimensions gap [**#3345259**](https://www.drupal.org/project/drupal/issues/3345259)) was floated as something that **could** be early-hinted without responsive branching; limited applicability to **hero** imagery in real themes.
- **Tangent — video posters:** [**#2954834 — Add poster image to HTML5 media videos**](https://www.drupal.org/project/drupal/issues/2954834) mentioned as **RTBC** in thread (poster as another late-discovered image class over time).

## Use

When designing **103** support in core or contrib: treat **image preload** (**#3309016** / **MR !14009**) and **responsive markup** as **the same constraint class** as **head `<link rel="preload">`**—early hints **do not magically** solve **art direction** or **`sizes: auto`** without **EARLY** knowledge of **layout inputs**. Prefer **CSS**/**font** early hints first; consider **`fetchpriority`** and **non-responsive** assets (e.g. **logo**) where URLs are **known before** **HTML** finishes.
